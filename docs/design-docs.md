# 中亚胡杨林生态系统保护数据共享平台软件设计文档

## 1. 文档概述

### 1.1 文档目的

本文档用于描述“中亚胡杨林生态系统保护数据共享平台”的软件总体设计、功能模块、页面组成、数据管理、权限控制、技术栈、配置方式、测试验收和交付要求。

本文档重点说明软件的功能设计和主要技术约束，不展开详细接口设计、数据库表结构设计、部署方案或底层代码实现细节。

### 1.2 软件定位

中亚胡杨林生态系统保护数据共享平台是一套 B/S 架构的网页端软件，面向胡杨林生态系统保护、空间数据管理、地图展示、数据查询、成果展示和数据共享服务。软件应该是中文界面。

用户通过浏览器访问系统。系统采用统一登录入口，登录后进入软件主界面。后台管理不作为登录页中的独立入口，而是作为登录后的功能模块，根据用户权限显示或启用。

### 1.3 建设目标与技术要求

#### 1.3.1 技术目标

中国胡杨林生态系统保护数据共享平台开发的总体目标是建设一个面向胡杨林生态系统保护的综合性数据共享平台，实现胡杨林生态系统多源异构数据的高效管理与可视化展示。重点关注遥感影像数据和胡杨林生态系统保护过程中实地调查与长期监测形成的野外数据的综合展示与利用，为胡杨林生态系统保护提供可视化的数据底座与业务应用支撑。

#### 1.3.2 技术内容

**（1）建立统一的数据资源管理体系**

本平台的开发首先将建立一套面向胡杨林生态系统保护的统一的标准化数据资源管理体系。实现从多源异构数据接入、野外非标准化数据规范指引到数据资源统一建库与归档的全流程操作。平台需具备对主流的空间矢量数据格式，高分遥感栅格影像以及实地调查和长期监测数据等多源异构数据的解析与接入能力。通过平台前端交互功能支持数据专题类型定义及时空范围确定等功能。针对格式不统一的野外调查和监测数据，构建前置标准化合规检查机制，提供覆盖核心业务专题的标准化数据导入模板和编制指南。在数据导入提交时，支持自动扫描校验数据结构、字段类型及空间边界的合理性，输出结构化报错原因。通过检查的数据自动存储至平台数据库，为上层应用提供结构化、标准化的数据支撑。

**（2）实现多源异构数据的集成展示**

平台将构建基于虚拟三维地球的数据集成和可视化展示系统，实现多尺度、多来源、多专题生态保护数据的展示。平台基于前后端分离架构，通过三维地球引擎展示多源数据，展示胡杨林分布区的地理生态环境。平台提供树状的图层管理器，支持图层的叠加以及显示状态控制，并提供测量地面距离与表面积的空间量算工具。此外，平台也提供标准化的符号系统，对重要要素设计固定的具备辨识度的图标，并根据影像缩放自动调整符号大小。针对已经构建影像金字塔的超大规模遥感影像，平台直接接入标准的网络地图瓦片服务解析瓦片数据。此外，平台将内置标准化生态专题制图模板。

#### 1.3.3 技术方法和路线

平台采用基于前后端分离的浏览器/服务器（B/S）技术架构，作为整个平台空间数据展示的运行架构。

**前端交互与三维展示：**
前端交互界面的搭建过程中，建立虚拟三维地球引擎，通过无插件的WebGL技术直接加载各类多源异构数据。为保障多尺度浏览的流畅度，平台建立细节层次控制（LOD）机制，支持场景从大范围宏观视角、区域流域中观视角到局部细节的微观视角的无缝平滑缩放。平台前端采用具有科技看板风格的交互式布局，所有侧边面板、浮动窗体以及图例组件均支持独立折叠。此外平台部署树状图层管理器，支持图层组多层叠加、显示状态控制显隐勾选、透明度滑块调节、图层上下覆盖顺序拖拽调整以及一键缩放至当前图层范围。同时，平台实现基础的空间量算工具，支持通过鼠标交互动态计算地面距离与多边形表面积。平台设置网格化属性表查看器以展示矢量图层的完整属性字段，实现空间几何图形与属性数据行之间的对应联系。在平台运维与安全保障方面，部署区分不同角色的多级访问控制系统与关键操作日志记录模块，并开发全局加载状态提示动画与数据导入报错机制，确保发生异常文件读取或网络断开时能够弹出文字提示。

**多源异构数据的存、管、查：**
针对多源异构数据，平台自动进行格式规范校验和数据资源统一建库与归档。在数据导入阶段，平台搭建统一的数据接入交互配置模块作为控制台，不仅支持对主流空间矢量数据及高分遥感栅格影像的直接导入，还能够自动提取实地调查与监测数据的地理空间坐标字段，并转化为三维空间点位要素。可通过交互配置模块手动或自动完成专题类型定义及时空范围划定。针对野外调查和监测数据，平台设置前置检查机制，统一向用户下发标准化数据导入规范。导入提交时自动化扫描确认文件的数据结构、字段类型及空间边界合理性，输出精准的报错原因引导用户修正。合规数据存入空间数据库统一建库与归档。平台基于“专题-时间-空间-来源”的多维分类逻辑构建统一的数据资产目录，入库时自动分类并归档。平台集成关键词文本、分类下拉菜单等多条件组合检索方式。

**可视化功能与制图：**
平台采用标准化、固定化的符号系统、颜色渲染机制及专题制图模块。在符号系统方面，针对离散点要素设计专用的图标与地标模型，并结合主视窗缩放机制实现符号大小自动调整；针对线、面要素提供线宽、描边颜色及内部填充色透明度的自定义配置。色彩渲染方面，针对连续型栅格影像与离散型矢量数据分别内置多套色带与分类色表，支持基于属性字段渲染。平台提供标准化的生态专题图模板，实现图层自动组合与图例数值比例尺的智能同步。对于超大规模遥感影像，直接接入标准网络地图瓦片服务拉取瓦片数据，前端界面按主视窗缩放大小实施按需加载。

### 1.4 设计边界

本软件包含以下内容：

1. 网页端地图展示与业务功能。
2. 登录后的后台管理功能。
3. 数据资源、数据目录、图层、成果和元信息管理。
4. 用户、角色和权限管理。
5. 矢量数据前端符号化展示。
6. 栅格数据后台符号化 XYZ 瓦片渲染和展示。
7. TOML 配置文件加载和关键配置校验。
8. 系统运行日志和关键操作记录。

本阶段不包含以下内容：

1. 原始遥感影像的专业生产处理流程。
2. 大规模遥感算法模型训练。
3. 独立移动端 App。
4. 第三方大型数据中心建设。
5. 复杂办公审批流系统。
6. 专题图制作功能。
7. 用户自定义样式模板功能。

专题图制作和用户自定义样式模板作为后续扩展能力预留，不作为本阶段必须交付内容。

## 2. 总体设计

### 2.1 软件形态

本软件采用 B/S 架构。用户通过浏览器访问系统，完成登录后进入软件主界面。

软件主界面承载业务功能和后台入口。后台入口作为软件主界面中的菜单项或功能按钮，根据用户权限显示或启用。

### 2.2 页面入口关系

系统页面入口关系如下：

1. 用户访问系统地址。
2. 系统展示统一登录页。
3. 用户登录成功后进入软件主界面。
4. 普通用户使用地图浏览、图层管理、数据查询、成果展示等业务功能。
5. 具备后台权限的用户可在软件主界面中进入后台管理功能。
6. 无后台权限的用户不显示后台入口，或入口不可点击并提示无权限。

### 2.3 用户角色

系统用户角色包括：

1. 普通用户：使用地图浏览、图层查看、数据查询、属性查看和成果浏览等基础功能。
2. 科研用户：在普通用户功能基础上，可使用更高权限的数据查询、结果导出、数据元信息查看等功能。
3. 数据管理员：负责数据资源、图层、目录、成果资料和数据元信息维护。
4. 系统管理员：负责用户、角色、权限、系统配置、日志和后台管理功能维护。

如需公开访问能力，可根据项目需要配置游客或公开用户权限。

### 2.4 权限原则

系统应根据用户角色和权限控制页面、菜单、按钮、图层、数据资源和后台功能访问范围。

权限控制原则如下：

1. 未登录用户访问受保护页面时跳转到统一登录页。
2. 登录用户只能访问已授权的功能和数据。
3. 无后台权限用户不得进入后台管理界面。
4. 无权限菜单不显示或不可点击。
5. 无权限图层不显示或不可加载。
6. 无权限数据不返回查询结果。
7. 数据导出、数据维护、系统配置等敏感操作必须进行权限校验。

## 3. 软件主界面设计

### 3.1 登录页

系统应提供统一登录页。登录页不展示独立后台入口。

登录页应包含：

1. 用户账号输入。
2. 用户密码输入。
3. 登录按钮。
4. 登录状态提示。
5. 记住登录状态选项。
6. 忘记密码入口。
7. 用户注册入口，是否开放注册由后台配置决定。

### 3.2 软件主界面

用户登录成功后进入软件主界面。软件主界面是业务功能和后台入口的统一承载页面。

软件主界面应包含：

1. 顶部导航栏。
2. 地图主视图。
3. 图层管理面板。
4. 数据查询面板。
5. 工具栏。
6. 结果展示面板。
7. 图层渲染与可视化面板。
8. 用户信息与退出入口。
9. 后台管理入口，是否显示或启用由用户权限决定。

### 3.3 后台管理界面

具备后台权限的用户从软件主界面进入后台管理界面。

后台管理界面应包含：

1. 后台顶部栏。
2. 左侧菜单栏。
3. 主内容区。
4. 用户信息区。
5. 系统消息与操作反馈区。

后台菜单应根据用户角色和权限动态加载。

## 4. 软件功能设计

### 4.1 地图浏览与空间量算

地图浏览是软件主界面的核心功能。系统基于虚拟三维地球引擎支持用户在浏览器中浏览项目相关各种空间数据。

地图浏览与操作应支持：

1. 宏观、中观到微观视角的无缝平滑缩放，基于细节层次控制（LOD）加载。
2. 地图放大、缩小、平移、旋转和倾斜浏览。
3. 比例尺机制和经纬度坐标显示。
4. 全屏显示与视角复位。
5. 按指定区域或坐标定位。
6. 底图切换，支持超大规模遥感影像按需加载的瓦片服务。
7. 多图层叠加显示及其交互控制。
8. 图层透明度调节。
9. 通过鼠标交互进行地面距离与多边形表面积计算的空间量算工具。
10. 全局加载状态提示动画，在异常读取或网络断开时提供文字提示。

### 4.2 树状图层管理

图层管理用于对地图中加载的数据图层进行统一管控，以树状图层管理器形式提供。

系统应支持：

1. 树状层级的图层列表查看与分类展示。
2. 图层组的多层叠加与合并管理。
3. 显隐勾选：控制图层的显示状态。
4. 透明度滑块调节。
5. 图层上下覆盖顺序的拖拽调整。
6. 一键缩放至当前图层范围（图层快速定位）。
7. 图层开启和关闭。
8. 图层名称、图例与元信息显示。
9. 支持图例组件的独立折叠。
10. 图层搜索。

图层分类应支持后台配置，不在程序中固定写死。图层操作响应时间目标不超过 500ms，系统应支持不少于 15 个图层并发加载与管理。

### 4.3 数据加载

数据加载用于将平台已登记的数据资源加载到地图中展示。

系统应支持：

1. 从数据目录选择数据加载。
2. 按图层分类加载数据。
3. 按空间范围或项目范围加载数据。
4. 按时间范围加载数据。
5. 批量加载多个图层。
6. 取消已加载数据。
7. 查看数据加载状态。
8. 加载失败提示。
9. 重新加载数据。

具体数据类型、字段结构、坐标信息、空间范围和展示方式以后续实际数据清单为准。

### 4.4 数据查询与多条件检索

数据查询用于帮助用户从平台数据资源中结构化、标准化地检索数据对象、属性信息和空间范围。

属性查询应支持：

1. 关键词文本查询。
2. 专题分类下拉菜单等分类查询方式。
3. 按数据来源查询。
4. 按时间信息查询。
5. 按字段条件或自定义条件查询。

空间查询应支持：

1. 点选查询。
2. 矩形范围查询。
3. 多边形范围查询。
4. 圆形缓冲区查询。
5. 当前地图范围查询。
6. 指定空间范围查询。

组合查询应支持关键词文本、专题分类下拉菜单、空间范围、时间条件等多条件组合检索方式。

查询结果应支持：

1. 地图高亮显示。
2. 列表展示。
3. 表格展示。
4. 弹窗展示。
5. 详情页展示。
6. 统计数量展示。
7. 授权范围内的结果导出。

当单次查询数据量不超过 3 万条时，查询响应时间目标不超过 1 秒。

### 4.5 网格化属性表查看与详情展示

系统采用网格化属性表查看器以展示矢量图层的完整属性字段，实现空间几何图形与属性数据行之间的双向联动和对应联系。用户点击地图对象或在属性表中选择记录时，系统应展示联动结果或详情。

详情展示内容应包括：

1. 对象名称。
2. 对象类型。
3. 所属数据集。
4. 主要属性字段。
5. 空间位置信息。
6. 更新时间。
7. 数据来源。
8. 数据质量说明。
9. 关联图层。
10. 关联附件或图片。
11. 返回地图定位。

### 4.6 矢量符号化渲染

矢量数据符号化由前端基于三维地球引擎进行渲染表达。

系统应支持：

1. 离散点要素采用专用的图标与地标模型进行展现。
2. 结合主视窗缩放机制实现点符号图标大小随底图自适应调整。
3. 线要素提供自定义配置线宽、描边颜色。
4. 面要素提供自定义配置描边和内部填充色透明度设置。
5. 针对离散型矢量数据提供内置多套标准生态分类色表。
6. 支持基于属性字段进行分类、分级渲染。
7. 标签显示与标签字段选择。
8. 结合生态专题图模板进行自动组合调用。

### 4.7 遥感影像与栅格渲染展示

遥感影像数据、连续型栅格以及后台出图结果的展示能力。

系统应支持：

1. 针对超大规模的遥感影像，支持直接接入标准的网络地图瓦片服务。
2. 根据客户端主视窗的缩放和范围自动解析并按需加载瓦片数据。
3. 支持连续型栅格影像内置多套色带进行设色渲染。
4. 加载后台专题脚本生成的动态 PNG 图像。
5. 设置遥感影像或栅格图层透明度。
6. 查看栅格图层图例和说明，并支持图例数值比例尺智能同步。
7. 查看栅格和影像元信息。
8. 处理图片或服务加载失败并向用户给出文字提示。

### 4.8 数据目录浏览

数据目录用于组织和展示平台已接入的数据资源。

系统应支持：

1. 按数据分类浏览。
2. 按业务主题浏览。
3. 按数据来源浏览。
4. 按更新时间浏览。
5. 按空间范围或项目范围浏览。
6. 搜索数据资源。
7. 查看数据简介。
8. 查看数据元信息。
9. 加载数据到地图。
10. 查看数据权限状态。

数据分类、业务主题和空间范围应支持后台配置。

### 4.9 成果展示

成果展示用于集中展示项目形成的图件成果、数据成果、说明材料和平台应用成果。

系统应支持：

1. 成果列表展示。
2. 成果分类筛选。
3. 成果详情查看。
4. 成果地图定位。
5. 成果图层加载。
6. 成果附件查看。
7. 成果图片预览。
8. 成果说明文本展示。
9. 成果更新时间展示。
10. 成果来源展示。

### 4.10 统计分析展示

系统应提供基础统计展示能力，用于对查询结果、图层数据和成果数据进行概览展示。

系统可支持：

1. 数据总量统计。
2. 分类数量统计。
3. 区域分布统计。
4. 时间分布统计。
5. 查询结果统计。
6. 图表展示。
7. 统计结果与地图联动。

图表类型可包括柱状图、折线图、饼图、环形图、指标卡片和简单排行表。

### 4.11 个人中心

3. 所有的矢量地理数据统一共用一个 GeoPackage 数据库，文件格式为 `.gpkg`，存放在地理数据总目录下的固定矢量数据目录中。
4. 栅格源文件以文件形式直接存储，存放在地理数据总目录 `raster/` 中。

6. `gdalinfo -json` 元数据存放在地理数据总目录 `metadata/` 中。

系统不得将业务附件、数据库文件、GeoPackage、原始栅格文件或栅格预处理文件与程序代码混合存放。

## 7. 栅格符号化与瓦片设计

### 7.1 处理原则

栅格文件不在前端进行符号化处理。系统应在后台读取栅格文件，并根据符号化规则实时生成 XYZ PNG 瓦片后提供给前端展示。

### 7.2 处理流程

栅格出图流程如下：

1. 栅格源文件保存到地理数据根目录 `raster/`。
2. 后台导入或启动扫描发现未处理源文件后，执行 `gdalinfo -json` 记录源文件元数据。
3. 后台执行 `gdalwarp -t_srs EPSG:3857 -r nearest -co COMPRESS=DEFLATE -of COG "$in" "$out"`，将预处理文件保存到 `preprocessed/`。
4. 后台再次执行 `gdalinfo -json` 记录预处理文件元数据，并记录导入时间、处理日志、源文件路径、预处理路径、文件大小、波段数、空间范围和默认符号化规则。
5. 前端请求栅格图层展示或重新设定符号化。
6. 后台根据预处理 COG 文件和符号化规则生成样式哈希。
7. 后台向前端返回 XYZ 瓦片 URL 模板、范围信息、样式哈希和实际规则。
8. 前端使用 XYZ 瓦片在地图中展示栅格图层。
9. 瓦片请求到达后，后台通过 Rasterio windowed read 读取对应窗口并实时应用符号化规则。

### 7.3 专题脚本机制

栅格文件符号化通过专用 Python 脚本实现。每个专题栅格文件可配置单独的符号化脚本。

系统应预留统一调用接口，用于屏蔽不同专题脚本之间的差异。统一接口应支持标准输入和标准输出，便于后台统一调度、日志记录、错误处理和瓦片渲染。

统一输入建议包括：

1. 栅格文件路径。
2. 输出瓦片范围或窗口参数。
3. 符号化规则配置。
4. 输出瓦片尺寸。
5. 数据范围或裁剪范围。
6. 透明背景配置。
7. 其他专题脚本扩展参数。

统一输出建议包括：

1. 执行状态。
2. 输出瓦片字节或临时结果。
3. 输出瓦片宽度和高度。
4. 栅格值范围或统计信息。
5. 错误信息。
6. 执行耗时。

### 7.4 瓦片样式机制

栅格加载统一使用 XYZ 动态瓦片，不生成整图 PNG 文件或缓存。

瓦片样式机制应支持：

1. 根据栅格文件和符号化规则生成样式哈希。
2. 在服务端登记样式哈希与归一化符号化规则的对应关系。
3. 前端使用统一 XYZ URL 模板加载瓦片。
4. 瓦片接口按数据集 ID、样式哈希、z/x/y 读取对应 COG 窗口。
5. 瓦片接口实时输出 PNG tile 字节，不保存整图 PNG 文件。

### 7.5 栅格符号化规则

本阶段栅格符号化由后端完成，前端只负责配置规则、显示进度和加载结果。系统支持以下规则：

1. 单波段灰度显示，支持 min/max 拉伸。
2. 任意三个波段组合为 RGB，支持逐波段 min/max 拉伸，并支持通过 mask 或指定波段配置 A 透明度。
3. 单波段伪彩色，支持内置色带和拉伸。
4. 单波段唯一值渲染，支持值到颜色的映射表；唯一值仅适用于整型波段，浮点型波段不适用。

默认符号化方案如下：

1. 1 个波段：单波段灰度，启用拉伸。
2. 2 个波段：第 1 波段映射 R，第 2 波段映射 G 和 B，启用拉伸。
3. 3 个及以上波段：取前 3 个波段映射 RGB，启用拉伸。

符号化参数优先使用预处理文件 `gdalinfo -json` 中记录的波段统计、尺寸、范围和坐标信息；当预处理文件缺少统计值时，可回退使用源文件 `gdalinfo -json` 的统计值生成默认规则。

唯一值不作为导入阶段的预统计内容。用户在前端选择整型波段并点击“分类”后，后台按块读取栅格并即时生成唯一值列表，不统计每个值的数量，避免整图一次性读入内存。

RGB 透明度规则应支持 `alphaBand` 参数：`mask` 表示使用 GDAL mask，数值表示指定波段作为透明度来源，空值表示不单独指定 A。XYZ 瓦片通过 Rasterio masked read 保持 nodata 透明。

前端应支持复制完整栅格符号化方案为 JSON。JSON 应包含透明度、渲染模式、波段选择、A 透明度、nodata、拉伸、色带和唯一值色表等完整内容。后台应支持为不同栅格数据或图层配置默认符号化方案，使无自定义符号化权限的用户也能按默认 XYZ 方案完整加载。

### 7.6 加载与进度通信

系统仅支持一种栅格加载方式：

1. XYZ 动态瓦片：后端为指定影像和符号化规则生成内存样式哈希，前端使用统一瓦片接口加载；瓦片服务通过 Rasterio windowed read 从 COG 的内部金字塔或合适分辨率窗口读取数据，所有影像共用同一个瓦片入口，通过数据集 ID 和样式哈希区分。

导入、预处理和重新符号化均通过异步任务接口与前端通信进度。`gdalwarp` 的命令行进度输出直接写入任务消息，前端在图层树或后台界面显示最近进度。

## 8. TOML 配置文件设计

### 8.1 配置原则

配置文件应体现程序、业务数据和地理数据分离的原则。程序目录不得作为业务数据总目录或地理数据总目录使用。

### 8.2 配置内容

1. 系统名称。
2. 运行模式。

4. 地理数据总目录。

7. 默认地图缩放级别。
8. 栅格符号化统一接口相关配置。
9. 其他后续扩展参数。

业务数据总目录和地理数据总目录下的子目录组织由系统固定，不在 TOML 配置文件中逐项配置。

### 8.3 配置校验

系统启动时应读取 TOML 配置文件，并对关键配置项进行校验。

校验内容包括：

1. 配置文件是否存在。
2. 配置格式是否正确。
3. 业务数据总目录是否有效。

5. 目录是否具备必要读写权限。
6. 缓存大小上限是否合理。
7. 固定子目录是否存在。
8. 栅格符号化统一接口配置是否有效。

固定子目录不存在时，系统可根据配置策略自动创建，或记录错误并提示管理员处理。

## 9. 技术栈说明

### 9.1 前端技术

前端采用以下技术：

1. React：用于构建网页端应用界面、组件状态管理和页面交互逻辑。
2. Vite：作为前端工程构建工具，用于项目开发、调试、构建和打包。
3. Mapbox GL JS：用于网页端三维全景地图展示、矢量图层渲染、多尺度 LOD 控制、空间交互及遥感影像加载。
4. Ant Design：用于实现除地图渲染外的主要 UI 组件，包括登录页、导航栏、侧边面板、浮动窗体、图例组件、表格、表单、后台管理页面和科技看板风格的交互式布局。
5. HTML、CSS、JavaScript 或 TypeScript：用于网页界面基础开发和工程组织。

前端负责多尺度地图展示、矢量图层分级符号化渲染、后台动态瓦片/图像展示以及页面 UI 交互（包含侧边面板与图例组件独立折叠功能）。前端界面设计应遵循前后端分离架构原则，并实现交互友好、容错清晰的界面逻辑。

### 9.2 后端技术

后端采用以下技术：

1. Python：作为后端主要开发语言。
2. Django：作为后端 Web 框架，用于实现业务管理、用户认证、权限控制、后台管理、数据接口和系统基础功能。
3. GeoPandas：用于矢量地理数据读取、处理、转换和空间数据分析支持。
4. GDAL：用于地理空间数据格式读取、转换和基础处理支持。
5. Rasterio：用于栅格数据读取、元信息提取和栅格文件处理支持。

后端负责业务管理、权限校验、数据资源管理、GeoPackage 读取、栅格文件读取、专题栅格脚本调度、XYZ 瓦片渲染和 TOML 配置加载。

### 9.3 Django 能力使用原则

在满足项目功能的前提下，应优先使用 Django 自带或 Django 生态中成熟稳定的能力实现基础功能，避免重复开发。

可优先使用 Django 能力的功能包括：

1. 用户账号管理。
2. 用户登录与退出。
3. 用户认证。
4. 用户组管理。
5. 角色与权限控制。
6. 后台管理页面基础能力。
7. 表单校验。
8. 数据模型管理。
9. 会话管理。
10. 基础安全防护能力。
11. 日志和管理操作记录。

项目特有功能，如 GeoPackage 矢量数据读取、栅格后台符号化瓦片渲染和专题脚本调度，可根据业务需要定制开发。

## 10. 非功能性设计

### 10.1 易用性

系统应具备良好的易用性，包括：

1. 页面结构清晰。
2. 菜单命名准确。
3. 常用功能入口明显。
4. 地图操作符合常见 WebGIS 使用习惯。
5. 查询条件填写简单明确。
6. 操作成功或失败应有明确提示。
7. 错误信息应便于用户理解。
8. 后台表单应具备必填项提示和格式校验。

### 10.2 兼容性

系统应支持主流现代浏览器访问，包括 Chrome、Edge、Firefox 和 Safari。

系统应优先保证桌面端浏览器体验，兼顾常见分辨率下的页面自适应展示。

### 10.3 性能

系统应满足以下性能目标：

1. 地图浏览帧率目标不低于 20fps。
2. 图层操作响应时间目标不超过 500ms。
3. 数据量不超过 3 万条时，单次查询响应时间目标不超过 1 秒。
4. 页面常用操作应具备及时反馈。
5. 大数据加载或栅格瓦片样式注册时应显示加载状态。
6. 栅格瓦片渲染应使用 COG 窗口读取，避免整图读入内存。

### 10.4 安全性

系统应具备基础安全能力，包括：

1. 用户登录认证。
2. 密码保护。
3. 权限校验。
4. 防止未授权访问后台。
5. 防止越权访问数据。
6. 关键操作记录日志。
7. 敏感操作二次确认。
8. 异常登录或失败操作提示。
9. 栅格脚本调用参数校验。
10. 配置目录读写权限校验。

### 10.5 可维护性和可扩展性

系统应支持后续维护和扩展，包括：

1. 后台可维护用户、角色和权限。
2. 后台可维护数据目录和图层配置。
3. 后台可维护成果资料。
4. 后台可查看操作日志和栅格脚本执行日志。
5. 数据资源应具备清晰元信息。
6. 可扩展新的数据类型、图层分类、统计图表、用户角色和权限项。
7. 可扩展新的专题栅格符号化脚本。
8. 可扩展外部数据服务接入。

### 11.2 鲁棒性测试

鲁棒性测试应覆盖：

1. 错误账号密码登录。

3. 空查询条件。
4. 无结果查询。
5. 数据加载失败。

压力测试应覆盖：

2. 大范围地图浏览。
3. 单次 3 万条以内数据查询。
4. 高频图层显隐切换。
5. 多用户同时访问基础功能。
6. 同一栅格文件多用户并发 XYZ 瓦片请求。

### 11.4 验收重点

验收重点包括：

1. 是否通过统一登录页进入系统。
2. 后台是否作为登录后的功能并按用户权限显示或启用。
3. 用户权限控制是否有效。
4. 地图浏览是否流畅。
5. 图层管理是否满足要求。
6. 数据查询是否正确。
7. 查询响应性能是否满足要求。
8. 矢量前端符号化渲染是否可用。
9. 栅格后台 XYZ 瓦片渲染和前端展示是否可用。
10. 栅格符号化规则和默认方案是否生效。
11. TOML 配置文件是否完整，业务数据总目录和地理数据总目录是否正确加载。

## 14. 总结

中亚胡杨林生态系统保护数据共享平台是一套面向生态保护、空间数据管理和科研成果共享的网页端软件。

系统采用统一登录入口，用户登录后进入软件主界面。后台作为登录后的一个功能模块，根据用户角色和权限显示或启用。

软件主界面侧重地图浏览、图层管理、数据查询、矢量前端符号化渲染、栅格后台 XYZ 瓦片展示和成果浏览。后台侧重用户权限、数据资源、图层配置、栅格符号化脚本配置、TOML 配置文件管理、成果发布和系统维护。

系统应将程序本身、业务数据和地理数据分离存放，通过 TOML 配置文件设定业务数据总目录和地理数据总目录，并在两个总目录下采用固定子目录组织。具体数据类型和数据清单应以后续实际资料为准。
